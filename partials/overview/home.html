Optimization Theory

Optimization theory is the language and toolset used to identify the optimal solution among all candidates for a given problem. Modeling a real-world problem using this language is an artform in and of itself. On SPConservation, we provide tools for solving optimization problems using a browser and the internet as a robust compute platform.
One method for solving an optimization problem is to implement a related signal processing system endowed with an appropriate conservation principle. You can learn more about this approach here. Specifically, for (most) optimization problems in the Examples library, two types of solvers are available:
  • Local solvers: signal processing systems organized into a single webpage for implementation. We provide four solvers for local systems.
  • Distributed solvers: signal processing systems organized into a database (Firebase) which can be implemented using any number of distributed workers. We provide two solvers for distributed systems.
Solving an optimization problem corresponds to running the associated signal processing system until it reaches a fixed-point. We provide both synchronous and asynchronous implementation tools to accomplish this.
Check out the Examples library to see various problems formulated using the language of optimization theory. Select one to solve the described problem by implementing the companion conservative signal processing structure yourself.

Signal Processing

Signal processing is the study of signals, systems, and their interactions. Signals are typicaly described using discrete or continuous functions while systems are described as the operators which process them. The applications of signal processing theory are endless; common ones include electrical engineering, seismology, audio processing, finance and economics.
On SPConservation, we focus on the design and implementation of conservative signal processing systems represented using signal-flow structures. The use of conservation is in linking the structure to an optimization problem in such a way that running the system to a fixed-point effectively solves the associated problem. We provide tools for two types of systems: local or non-distributed and distributed systems. For distributed systems, we decompose the signal-flow structure as well as the signal to be processed into objects and store them in a database so that a group of neworked workers can implement the system. For non-distributed systems, this process is performed completely within the browser

Signal-Flow Structures

Signal-flow structures provide a convenient way of organizing the variables associated with a system of equations into a computational form. The particular signal-flow structure depicted here is mathematically equivalent to all conservative signal processing systems implemented on this website. Signal-flow characterizations are behavioral in general and in this way can be thought of as a graphical declarative programming language. We provide two implementation protocols for running such a system:

  • Synchronous protocols: a traditional implementation of a signal processing system akin to iterating a function representing the system where the system state or memory acts as a vector delay unit.
  • Asynchronous protocols: the implementation of a signal processing system where the system state or memory acts as randomly triggered sample-and-hold elements.
A detailed treatement of these protocols can be found here.

Signal Processing + Conservation

Conservation principles applied to optimization theory are used to define a primal and dual pair of optimization problems whose collective constraints or feasibility conditions serve as sufficient conditions for stationarity. Further, the conditions associated with this class of optimization problems can always be cast into the conservative signal-flow structure depicted on the top right. Local Controls

The controls available for a local solver are organized into the local command center depicted above and operate as follows:
  • Initialize: Performs the precompute associated with transforming the problem parameters from the setup page into the associated conservative signal-flow structure.
  • Start: The signal processing system starts continuously processing.
  • Step: The system state updates once in accordance with the randomness parameter.
  • Stop: The signal-flow structure stops continuous processing.
  • Reset: The system state is reset to initialization according to the selected solver as described here.
  • Randomness: A slider taking values from zero (left) to one (right) corresponding to the probability each delay element updates for each iteration.
In addition, you can select one of the four local solvers from the settings menu.

Distributed Controls

The controls available for a distributed solver are organized into a Distributed Solver Command Center as depicted below for an example. The controls are organized as follows:
  • Worker Info: Metadata associated with registering networked worker resources
  • Worker Controls: Actions used to control the distributed worker resources.
  • Solver Settings: Metadata to control the solvers implemented by worker resources.
  • Visualization: The currently selected visualization plot updated according to the refresh rate.
  • Visualization Settings: Possible visualizations related to the solution and worker resources. Note that turning off or slowing the refresh rate decreases the overall bandwidth from the database allowing the workers to process faster.
